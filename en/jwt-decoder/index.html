JWT Structure Explained

A JWT (JSON Web Token) consists of three parts separated by dots ., in the format Header.Payload.Signature

1. Header

Typically contains two fields: alg (the signing algorithm, e.g., HS256, RS256) and typ (the token type, usually JWT). The Header is Base64URL-encoded.

Example: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

2. Payload

Contains claims, which are statements about an entity and additional data. There are three types:

  • Registered claims: iss (issuer), sub (subject), aud (audience), exp (expiration), nbf (not before), iat (issued at), jti (JWT ID)
  • Private claims: Custom claims agreed upon by both parties, e.g., name, admin
  • Public claims: Claim names registered in the IANA JWT Registry

The Payload is also Base64URL-encoded. Note: Base64URL encoding is not encryption — anyone can decode and read it.

3. Signature

Created using the algorithm specified in the Header, signing the Base64URL-encoded Header and Payload with a secret key. The signature verifies that the token has not been tampered with.

Process (HS256): HMAC-SHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

JWT Security Best Practices

  • Do not store sensitive data: The Payload can be decoded by anyone — never put passwords, SSNs, or other sensitive data in it
  • Always use HTTPS: JWTs can be replayed if intercepted in transit; HTTPS is a baseline requirement
  • Reject alg:none: Never accept tokens with alg:none, which means no signature verification
  • Set reasonable expiration: The exp field is mandatory; recommend short-lived tokens (15min–2h) with Refresh Tokens
  • Strong keys: Use at least 256-bit keys for HS256; avoid weak or hardcoded keys
  • Validate all fields: The server must verify alg, exp, iss, aud, and other claims
  • Storage location: Prefer HttpOnly Cookies to prevent XSS from stealing tokens
  • Consider revocation: Implement token blacklists or version-based mechanisms for active invalidation

What is JWT and what is it used for?

JWT (JSON Web Token) is an open standard (RFC 7519) for securely transmitting information between parties as a JSON object. It is commonly used for authentication, information exchange, and single sign-on (SSO). After a server issues a JWT, the client includes it in subsequent requests to prove identity, without the server needing to store session state.

What information can be seen after decoding a JWT?

A JWT consists of three parts: Header (containing algorithm and type), Payload (containing claims such as user ID, issue time, expiration time), and Signature. The Header and Payload are Base64URL-encoded and can be decoded by anyone. Never put passwords or sensitive data in a JWT. The signature is used to verify the token has not been tampered with, but this tool does not verify signatures.

Does this JWT decoder verify the signature?

No. This tool only decodes and displays the Header and Payload content of a JWT. It does not verify the signature. Verifying a signature requires a secret key (HS256) or public key (RS256/ES256), and this is a pure frontend zero-dependency decoder that does not accept any key input. To verify signatures, use services like jwt.io or verify in your server-side code.

How do I read the exp expiration time in a JWT?

The exp field in a JWT is a Unix timestamp (seconds since January 1, 1970) representing the token's expiration time. This tool automatically detects the exp field, converts it to a human-readable date, and calculates the remaining valid time or how long it has been expired. If the token has no exp field, it means no expiration was set, which is not recommended in production.

What is the difference between HS256 and RS256 algorithms?

HS256 is a symmetric algorithm that uses the same secret key for both signing and verification, suitable for single-service scenarios. RS256 is an asymmetric algorithm that uses a private key to sign and a public key to verify, suitable for microservice architectures and multi-party verification. ES256 uses elliptic curve cryptography with shorter keys but comparable performance. The choice depends on performance, key management, and trust model requirements.

Is JWT secure? What precautions should I take when using it?

JWT is a secure transmission mechanism, but improper use can introduce risks. Precautions include: do not store sensitive information in the Payload (Base64URL encoding is not encryption); always use HTTPS; never accept alg:none tokens; set reasonable expiration times; use sufficiently strong keys for HS256; implement token refresh mechanisms and blacklists for active revocation.
