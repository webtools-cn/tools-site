📖 Semantic Versioning (SemVer) Guide

Version Format

Semantic versions follow MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD] format:

  • MAJOR: Incremented for incompatible API changes
  • MINOR: Incremented for backward-compatible new features
  • PATCH: Incremented for backward-compatible bug fixes
  • PRERELEASE: Optional, e.g., -alpha.1, -beta.2
  • BUILD: Optional, e.g., +build.123, does not affect precedence

Precedence Rules

  1. Compare MAJOR first, higher number has higher precedence
  2. If MAJOR equal, compare MINOR
  3. If MINOR equal, compare PATCH
  4. Release version beats pre-release (1.0.0 > 1.0.0-alpha)
  5. Pre-release identifiers compared segment by segment: numeric as integers, alphanumeric lexicographically
  6. Build metadata does not affect precedence

Common Mistakes

  • Leading zeros: 01.2.3 ❌ (should be 1.2.3)
  • Missing parts: 1.2 ❌ (should be 1.2.0)
  • Empty identifiers: 1.2.3-. ❌
  • Invalid characters: 1.2.3-α ❌ (only ASCII alphanumeric and hyphens allowed)

📖 Frequently Asked Questions

What is Semantic Versioning (SemVer)?

Semantic Versioning (SemVer) is a versioning scheme using MAJOR.MINOR.PATCH format. MAJOR indicates incompatible API changes, MINOR adds backward-compatible functionality, and PATCH includes backward-compatible bug fixes. Optional pre-release identifiers (e.g., -alpha.1) and build metadata (e.g., +001) can be appended.

What are the SemVer 2.0 validation rules?

SemVer 2.0 requires: 1) Version format X.Y.Z where X, Y, Z are non-negative integers without leading zeros; 2) Pre-release identifiers consist of alphanumeric characters and hyphens, dot-separated; 3) Build metadata consists of alphanumeric characters and hyphens, dot-separated; 4) Pre-release versions have lower precedence than release versions; 5) Version numbers must not skip increments.

How do you compare two semantic versions?

Comparison rules: 1) Compare MAJOR first, higher number wins; 2) If MAJOR equal, compare MINOR; 3) If MINOR equal, compare PATCH; 4) Release version beats pre-release (1.0.0 > 1.0.0-alpha); 5) Pre-release identifiers are compared segment by segment: numeric identifiers compared as integers, alphanumeric compared lexicographically, numeric identifiers have lower precedence than alphanumeric.

What are pre-release identifiers and build metadata?

Pre-release identifiers follow PATCH with a hyphen, e.g., 1.0.0-alpha.1, indicating an unstable version. Build metadata follows with a plus sign, e.g., 1.0.0+201****4700, providing build information but not affecting version precedence. Both are optional.

Is my data uploaded to any server?

No. All version validation, parsing, and comparison is done locally in your browser. No data is sent to any server.
