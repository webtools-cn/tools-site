📖 Kubernetes YAML Validation Guide

Required Fields for All Resources

Every Kubernetes resource must have these top-level fields:

  • apiVersion: The API version (e.g., v1, apps/v1, networking.k8s.io/v1)
  • kind: The resource type (e.g., Pod, Deployment, Service)
  • metadata: Object containing at minimum a name field
  • spec: The resource specification (structure varies by kind)

Common Validation Errors

  • Missing required fields (apiVersion, kind, metadata.name)
  • Invalid apiVersion for the specified kind
  • Missing container name or image in Pod specs
  • Invalid port numbers (must be 1-65535)
  • Incorrect indentation (YAML is indentation-sensitive)
  • Duplicate keys in the same mapping

Supported Resource Types

Pod, Deployment, Service, ConfigMap, Secret, Ingress, PersistentVolumeClaim, Namespace, Job, CronJob, DaemonSet, StatefulSet, ReplicaSet, Role, RoleBinding, ClusterRole, ClusterRoleBinding, NetworkPolicy, HorizontalPodAutoscaler, and more.

📖 Frequently Asked Questions

What is Kubernetes YAML validation?

Kubernetes YAML validation checks that your K8s manifest files conform to the expected schema for each resource type. It verifies required fields, correct value types, valid API versions, and proper structure before you apply them to your cluster.

What are common Kubernetes YAML errors?

Common errors include: missing required fields (apiVersion, kind, metadata, spec), invalid apiVersion strings, incorrect indentation, wrong field types (string vs number), missing container name or image, invalid port ranges, and duplicate keys in mappings.

What Kubernetes resources does this validator support?

This validator supports common K8s resources including Pod, Deployment, Service, ConfigMap, Secret, Ingress, PersistentVolumeClaim, Namespace, Job, CronJob, DaemonSet, StatefulSet, ReplicaSet, Role, RoleBinding, ClusterRole, and ClusterRoleBinding.

How is this different from kubectl apply --dry-run?

kubectl dry-run validates against a real cluster's API server, which requires cluster access and considers CRDs and admission webhooks. This tool validates schema structure offline in your browser, which is faster and works without cluster access, but does not check cluster-specific constraints.

Can I validate multi-document YAML files?

Yes. This validator supports multi-document YAML files separated by --- (the standard Kubernetes manifest format). Each document is validated independently, and you get results for each resource in the file.

Is my YAML data sent to any server?

No. All YAML parsing and validation is done locally in your browser. No data is sent to any server. Your Kubernetes manifests remain completely private.
