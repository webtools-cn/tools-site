What can the Semver Tester do?

The Semver Tester is a free online tool that helps developers validate semantic version number (Semver) format, compare version order, and check if a version satisfies a specified range. Fully compliant with Semver 2.0.0 specification, it supports pre-release versions (alpha/beta/rc) and build metadata parsing and comparison.

Core Features

  • Version Validation: Check if version format complies with Semver 2.0 spec, parse MAJOR/MINOR/PATCH/pre-release/build metadata
  • Version Comparison: Compare two version numbers, show step-by-step difference analysis
  • Range Check: Verify if a version satisfies ^, ~, >= range constraints
  • Batch Validation: Support multi-line input for bulk version validation
  • Real-time Feedback: Validate as you type, no button click needed

How to Use

1. Enter a version number in the "Validate" tab, the tool automatically validates and parses each component

2. Switch to the "Compare" tab, enter two versions to see their ordering and differences

3. In the "Range Check" tab, enter a version and range expression to check constraint satisfaction

4. Click "Load Example" to quickly try each feature

Extended Knowledge

Semver 2.0.0 was created by Tom Preston-Werner and has become the de facto standard for software versioning. The format is MAJOR.MINOR.PATCH, where MAJOR changes indicate incompatible API modifications, MINOR changes indicate backward-compatible feature additions, and PATCH changes indicate backward-compatible bug fixes. Pre-release versions (e.g., -alpha, -beta) have lower precedence than normal versions, and build metadata (e.g., +build.123) does not participate in version comparison.

FAQ

Which is higher: 1.0.0-alpha or 1.0.0?▼
1.0.0 is higher. Pre-release versions (alpha/beta/rc) have lower precedence than the corresponding normal version, so 1.0.0-alpha < 1.0.0.
Does build metadata (+build) affect version comparison?▼
No. Per the Semver spec, build metadata does not participate in version precedence comparison. 1.0.0+build.1 and 1.0.0+build.2 are considered the same version.
What are the special rules for 0.x.x versions?▼
When MAJOR is 0 (0.x.x), it indicates initial development. The API may change at any time, and even MINOR versions do not guarantee backward compatibility.
What is the precedence order for pre-release versions?▼
Compared alphabetically: alpha < beta < rc < release. Numeric identifiers are compared numerically, alphanumeric identifiers are compared by ASCII sort order.
What version range syntaxes are supported?▼
Supports ^ (compatible version), ~ (approximate version), >/>=/</<= (comparison operators), - (hyphen range), x (wildcard) and other npm standard range syntaxes.
