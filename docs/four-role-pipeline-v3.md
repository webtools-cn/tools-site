# 四角色协作流水线 v3.0 — 设计方案

> 核心目标：数量 × 质量 × 流量 = 变现
> 核心问题：2000+工具在线，但流量差。10%工具达到个税计算器水平=翻天
> 核心原则：开发不推送，测试才推送。存量优化优先于新增。

## 二、四角色定义（融合调度中枢+质检v7）

### 角色1：产品经理 (PM)
**不是从零选工具，而是基于存量做决策**

| 职责 | 具体工作 |
|:-----|:---------|
| 存量审计 | 扫描2120个工具，按流量潜力分级（A/B/C/D） |
| 竞品分析 | 对标同类工具站，找差距（功能/SEO/内容） |
| 优化决策 | 决定哪些工具优先优化、哪些该下线、哪些该新增 |
| 验收标准 | 每个任务输出Checklist，QA照着验收 |
| SEO策略 | 基于GSC数据，定关键词优化方向 |
| GEO策略 | 确保LLM能发现和推荐我们的工具 |

**PM的分级标准**：
- **A级（流量潜力大）**：搜索量大、竞品弱、我们有优势 → 优先优化到个税计算器水平
- **B级（有潜力）**：搜索量中等、需SEO/内容优化 → 批量提升
- **C级（低价值）**：搜索量小、但维护成本低 → 保持现状
- **D级（负资产）**：空壳/质量差/误导用户 → 下线或重做

### 角色2：UX设计师 (UX)
**保证风格统一 + SEO/GEO在设计和验收环节融合**

| 职责 | 具体工作 |
|:-----|:---------|
| 风格基线 | 维护统一设计规范（颜色/字体/间距/布局） |
| 布局审核 | 新工具/优化工具必须与基线一致 |
| SEO融合 | title/description/h1/Schema在设计阶段就定好 |
| GEO融合 | 确保页面内容对LLM友好（结构化、可提取） |
| 移动端 | 375px不崩、不溢出 |

**UX的SEO/GEO融合点**：
- title格式：`{工具名} - Free ToolBase`（20-60字符，含核心关键词）
- description：50-160字符，含"免费在线"+核心功能+适用场景
- h1：含emoji+工具名，与title呼应但不完全重复
- Schema：SoftwareApplication(必选) + FAQ(必选) + HowTo(可选)
- 页面结构：h1→功能区→说明→FAQ→HowTo→footer（LLM可顺序提取）

### 角色3：开发 (DEV)
**只写代码，不推送**

| 职责 | 具体工作 |
|:-----|:---------|
| 新工具开发 | 按PM需求+UX规范开发CN+EN版 |
| 存量优化 | 按PM优先级优化A级/B级工具 |
| 自检 | JS语法/交互≥3/无假评分/无unicode |
| 代码留在工作区 | **禁止git commit/push** |

### 角色4：测试 (QA)
**唯一推送权限，融合质检v7八门**

| 职责 | 具体工作 |
|:-----|:---------|
| 扫描工作区 | `git status`找未提交文件 |
| 逐项验收 | 按PM的Checklist + UX规范 + 质检八门 |
| 浏览器实测 | 功能/布局/移动端/中英文 |
| SEO/GEO验收 | title/description/Schema/canonical/hreflang |
| 站点同步 | sitemap/llms.txt/首页卡片 |
| 通过→推送 | git add + commit + push + 线上验证 |
| 不通过→打回 | 说明问题，DEV修，最多3轮 |

## 四、流水线流程

### 4.1 存量优化流程（优先级最高）

```
PM: 扫描存量 → 按流量潜力分级(A/B/C/D)
  → A级工具：逐个深度优化（功能+SEO+GEO+内容）
  → B级工具：批量SEO/内容提升
  → D级工具：下线或重做
  ↓
UX: 审核优化方案 → 确保风格统一+SEO/GEO规范
  ↓
DEV: 按优先级执行优化 → 代码留在工作区
  ↓
QA: 验收 → 通过→推送 → 线上验证
```

### 4.2 新工具开发流程

```
PM: 竞品分析+搜索量验证 → 确认有流量价值 → 定需求+验收标准
  ↓
UX: 定布局+SEO/GEO规范 → 确保与存量风格统一
  ↓
DEV: 开发CN+EN → 自检 → 代码留在工作区
  ↓
QA: 验收 → 通过→推送+更新sitemap/llms.txt/首页 → 线上验证
```

### 4.3 批量修复流程

```
PM: 从质检报告/GSC数据识别问题 → 定修复范围+优先级
  ↓
UX: 确认修复不影响风格
  ↓
DEV: 批量修复 → 代码留在工作区
  ↓
QA: 抽样验证(至少5页) → 全量验证关键项 → 通过→推送
```

## 五、与现有系统的融合

### 5.1 调度中枢（保留+升级）
- 保留：每10分钟周期、delegate_task派子agent、p0阻塞机制
- 升级：子agent按角色分配，串行执行PM→UX→DEV→QA
- 新增：QA角色独占推送权限

### 5.2 质检v7八门（保留+融合进QA）
- 门0-7全部保留，作为QA验收的Checklist
- 不再单独跑质检cron，QA每次验收=跑一遍八门
- 质检报告由QA产出，不再单独脚本

### 5.3 Cron调度
| 名称 | 调度 | 职责 |
|:-----|:-----|:-----|
| 工具站流水线 | `05,35 * * * *` | PM→UX→DEV→QA完整周期 |
| 全站巡检 | `0 6,18 * * *` | QA全量扫描，产出质量报告 |
| SEO巡检 | `0 3 * * *` | PM基于GSC数据优化 |
| 数据备份 | `50 0 * * *` | 脚本 |
