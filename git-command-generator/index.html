📚 使用教程

本Git命令生成器旨在帮助开发者和Git初学者快速生成准确的Git命令。以下是详细的使用指南：

基础操作

在左侧「Git操作」下拉菜单中选择需要执行的Git操作类型。选择后，下方会出现该操作对应的参数配置选项。根据您的实际需求填写或勾选参数，右侧会实时生成对应的Git命令。

提交代码：选择「commit - 提交更改」操作，在「提交信息」输入框中填写描述本次更改的说明文字。勾选「允许空提交」可以在没有文件改动的情况下创建提交（常用于CI/CD触发）。点击生成的命令可直接复制到终端执行。

推送代码：选择「push - 推送到远程」操作，填写远程仓库名称（默认为origin）和分支名。勾选「强制推送（--force）」可以覆盖远程历史（谨慎使用）。如果您是第一次推送该分支，工具会自动添加-u参数建立追踪关系。

分支管理：选择「branch - 分支管理」操作，支持创建新分支、删除分支、重命名分支和列出所有分支四种模式。创建分支时可选择是否同时切换到新分支（相当于git checkout -b的效果）。删除分支时工具会提示需要先合并或强制删除。

高级用法

变基操作：选择「rebase - 变基」操作，可以配置交互式变基（-i）、自动继续（--continue）等选项。变基时如果遇到冲突，修复后需要执行git rebase --continue。工具会提示您当前的变基状态和下一步操作。

暂存管理：选择「stash - 暂存」操作，支持保存当前改动、应用最近一次暂存、列出所有暂存和删除指定暂存。保存时可以添加描述信息，方便后续识别。应用暂存时可以选择是否从列表中删除（pop vs apply）。

Git命令生成器在多种场景下都能发挥重要作用：

Git初学者学习：对于刚接触Git的开发者，记忆各种命令和参数是一个挑战。通过可视化的参数配置界面，学习者可以直观地了解每个命令有哪些可选参数、它们的作用是什么。生成的命令可以直接复制到终端执行，降低学习曲线。配合教程中的解释，初学者能够快速建立对Git命令体系的认知。

快速生成复杂命令：即使是资深开发者，面对git rebase -i HEAD~5或git cherry-pick abc123..def456这样的复杂命令时，也难免需要查阅文档确认语法。本工具将常见的复杂命令参数可视化，几秒钟就能生成准确的命令，避免手误导致的操作错误。

团队协作规范：在团队开发中，统一的Git工作流至关重要。团队可以将本工具作为内部参考，确保所有成员使用一致的命令格式。特别是对于commit message规范、分支命名规范等，可以通过工具快速生成符合团队标准的命令。

面试准备：Git是程序员面试中的高频考点。使用本工具复习常用命令，配合FAQ中的深度解析，可以帮助面试者系统性地掌握Git的核心概念和常用操作，提高面试通过率。

💡 扩展知识

Git工作流：常见的工作流包括Git Flow、GitHub Flow和GitLab Flow。Git Flow使用master/main和develop两个长期分支，配合feature、release、hotfix等短期分支，适合发布周期明确的项目。GitHub Flow更轻量，所有改动都通过Pull Request合并到main分支，适合持续部署的项目。选择合适的工作流可以提高团队协作效率。

Commit Message规范：良好的提交信息是项目可维护性的基础。常用的规范有Conventional Commits，格式为type(scope): subject，其中type包括feat、fix、docs、style、refactor、test、chore等。规范的提交信息可以自动生成CHANGELOG，也方便代码审查和历史追溯。

Git Hooks：Git提供了多种钩子（hook）机制，可以在特定事件发生时执行自定义脚本。常用的有pre-commit（提交前执行lint检查）、commit-msg（提交信息格式校验）、pre-push（推送前运行测试）。结合Husky等工具，可以在团队中强制执行代码质量和提交规范。

Git内部原理：Git的核心是一个内容寻址文件系统。每次提交都会生成一个SHA-1哈希值，指向一个树对象（tree），树对象包含多个blob对象（文件内容）和其他树对象。理解这一原理有助于更好地理解Git的分支、合并、变基等操作的本质——它们本质上都是对指针的重新定向。

❓ 常见问题 (FAQ)

Git和GitHub有什么区别？

Git是分布式版本控制系统，用于在本地管理代码版本。GitHub是基于Git的代码托管平台，提供远程仓库托管、协作、CI/CD等服务。简单来说，Git是工具，GitHub是使用Git工具的平台。类似的平台还有GitLab、Gitee、Bitbucket等。

如何撤销上次提交？

撤销上次提交有多种方式：使用git commit --amend可以修改上次提交的message；使用git reset --soft HEAD~1可以撤销上次提交但保留改动到暂存区；使用git reset --hard HEAD~1可以彻底丢弃上次提交的改动（不可逆，谨慎使用）；使用git revert HEAD会创建一个新的提交来反向抵消上次提交的改动，这是最安全的撤销方式。

git pull和git fetch有什么区别？

git fetch只从远程下载最新的变更到本地远程跟踪分支（如origin/main），不会合并到当前分支，因此是安全的。你可以先用git log HEAD..origin/main查看远程有哪些变更，确认无误后再手动git merge origin/main。git pull = git fetch + git merge，会自动将远程变更合并到当前分支，如果存在冲突需要立即解决。

git rebase和git merge哪个更好？

没有绝对的好坏。merge保留完整的历史分支结构，适合需要完整追溯历史的场景，是团队协作中公共分支的首选。rebase将当前分支的提交在目标分支顶端重新应用，产生线性历史，看起来更整洁，适合个人feature分支在合并前整理历史。重要原则：不要对已经推送到公共仓库的分支执行rebase，这会重写公共历史。

如何解决合并冲突？

冲突发生时，Git会在冲突文件中标记<<<<<<< HEAD、=======和>>>>>>> branch-name。需要手动编辑文件保留需要的代码，删除冲突标记，然后执行git add .和git commit完成合并。建议使用VS Code、WebStorm等IDE提供的可视化冲突解决工具，可以更直观地查看和选择代码。在团队协作中，提前沟通、减小单次PR范围可以有效减少冲突。

git stash怎么用？

git stash将当前未提交的改动保存到堆栈中，使工作区恢复干净。常用命令：git stash push -m "描述"保存当前改动；git stash list查看列表；git stash pop恢复最近一次并删除；git stash apply恢复但不删除；git stash drop删除指定暂存。适合临时切换分支处理紧急任务的场景。

为什么git push被拒绝了？

常见原因有：1）远程有本地没有的更新，需要先git pull合并远程变更后再push；2）没有该分支的写权限；3）使用了--force-with-lease但远程已被其他人更新。解决方法：先执行git pull --rebase拉取远程最新代码并变基到当前分支顶部，解决可能的冲突后再push。

如何查看某文件的修改历史？

使用git log -p 文件名可以查看该文件的完整修改历史，包括每次提交的差异。git blame 文件名可以逐行显示每行代码最后是由谁、在什么时间修改的。git log --graph --oneline --all可以以图形化方式查看所有分支的提交历史。这些命令都可以通过本工具快速生成。
