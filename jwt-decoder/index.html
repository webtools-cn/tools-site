JWT 结构说明

JWT（JSON Web Token）由三部分组成，用点号 . 分隔，格式为 Header.Payload.Signature

1. Header（头部）

通常包含两个字段：alg（签名算法，如 HS256、RS256）和 typ（Token 类型，通常为 JWT）。Header 使用 Base64URL 编码。

示例：eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

2. Payload（载荷）

包含 claims（声明），即关于实体和附加数据的声明。分为三类：

  • 标准声明：iss（签发者）、sub（主题）、aud（受众）、exp（过期时间）、nbf（生效时间）、iat（签发时间）、jti（唯一ID）
  • 私有声明：双方约定的自定义声明，如 name、admin
  • 公共声明：可参考 IANA JWT Registry 注册的声明名

Payload 同样使用 Base64URL 编码，注意：Base64URL 编码不是加密，任何人都可以解码查看

3. Signature（签名）

使用 Header 中指定的算法，对 Base64URL 编码的 Header 和 Payload 用密钥进行签名。签名用于验证 Token 未被篡改。

示例流程（HS256）：HMAC-SHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

JWT 安全注意事项

  • 不要存储敏感信息：Payload 可被任何人解码，切勿放入密码、身份证号等敏感数据
  • 始终使用 HTTPS：JWT 在传输中如被截获可被重放，HTTPS 是基本要求
  • 拒绝 alg:none：永远不要接受 alg:none 的 Token，这表示无签名验证
  • 设置合理过期时间：exp 字段不可省略，建议短期有效（15分钟~2小时）配合 Refresh Token
  • 密钥强度：HS256 密钥至少 256 位，避免使用弱密钥或硬编码密钥
  • 验证所有字段：服务端必须验证 alg、exp、iss、aud 等声明
  • 存储位置：优先存储在 HttpOnly Cookie 中，避免 XSS 攻击窃取 Token
  • 考虑撤销机制：实现 Token 黑名单或版本号机制，支持主动失效

JWT是什么？有什么用？

JWT（JSON Web Token）是一种用于在网络应用间安全传输信息的开放标准（RFC 7519）。它以JSON格式编码信息，常用于身份认证、信息交换和单点登录（SSO）。服务器签发JWT后，客户端在后续请求中携带它来证明身份，无需服务器存储会话状态。

JWT解码后能看到什么信息？

JWT由三部分组成：Header（包含算法和类型）、Payload（包含claims声明，如用户ID、签发时间、过期时间等）和Signature（签名）。Header和Payload使用Base64URL编码，任何人都可以解码查看。因此绝不要在JWT中放置密码等敏感信息。签名部分用于验证Token未被篡改，但本工具不验证签名。

这个JWT解码器会验证签名吗？

不会。本工具仅解码和展示JWT的Header和Payload内容，不验证签名的有效性。验证签名需要密钥（HS256）或公钥（RS256/ES256），而本工具是纯前端零依赖的解码工具，不接收任何密钥输入。如需验证签名，请使用jwt.io等服务或在服务端代码中验证。

JWT的exp过期时间怎么看？

JWT的exp字段是一个Unix时间戳（自1970年1月1日以来的秒数），表示Token的过期时间。本工具会自动检测exp字段，将其转换为可读日期，并计算剩余有效时间或已过期时间。如果Token中没有exp字段，说明该Token没有设置过期时间，这在生产环境中是不推荐的。

HS256和RS256算法有什么区别？

HS256是对称算法，使用同一个密钥进行签名和验证，适合单一服务场景。RS256是非对称算法，使用私钥签名、公钥验证，适合微服务架构和多方验证场景。ES256使用椭圆曲线加密，密钥更短但性能相当。选择算法时需考虑性能、密钥管理和信任模型。

JWT安全吗？使用时有哪些注意事项？

JWT本身是一种安全的传输机制，但使用不当会引入风险。注意事项包括：不要在Payload中存储敏感信息（Base64URL编码不等于加密）；务必使用HTTPS传输；设置合理的过期时间；不要使用alg:none；对于HS256使用足够强度的密钥；考虑实现Token刷新机制和黑名单来支持主动撤销。
